Chapter 01
Agent 是什么？
它如何工作？

Agent 不是一次性问答，而是一个持续运行的"思考→行动→观察"循环。理解这个循环，是掌握所有 Agent 技术的基础。

Agent 的三要素
AGENT = LLM + TOOLS + LOOP
LLM —— 根据当前上下文决定下一步
Tools —— 执行操作（搜索/读文件/调API）
Loop —— 重复执行直到任务完成
ReAct 循环 — Agent 每一步在做什么
── 用户发来任务 ──
用户输入
↓
① Reason：LLM 思考 "我需要先查天气再做决策"
↓
② Act：调用工具 getWeather("北京")
↓
③ Observe：观察结果 { temp: 22, condition: "晴" }
↓ 循环回①（直到无需再调工具）
最终回答给用户
与之前学过的 generateText 的本质区别

❌ generateText + maxSteps（手动）

你之前的写法
// 每次调用都要重新配置
const r = await generateText({
  model: openai('gpt-4o'),
  tools: { ... },
  maxSteps: 5,
  messages,
})
// 不可复用，每处都要写

✅ ToolLoopAgent（推荐）

现代写法
// 定义一次，处处复用
const agent = new ToolLoopAgent({
  model: "openai/gpt-4o",
  tools: { ... },
})

await agent.generate({ prompt })
await agent.stream({ prompt })
重要认知：AI SDK 6 引入了 ToolLoopAgent 类作为官方推荐的 Agent 构建方式。它本质上是对 generateText 的封装，但提供了可复用性、内置的循环管理和更简洁的 API。

Chapter 02
ToolLoopAgent

AI SDK 6 的核心新增类。定义一次，可在整个应用中复用。支持 generate（等待完整结果）和 stream（流式输出）两种模式。

最小完整示例
安装
npm install ai @ai-sdk/openai zod
定义 Agent
import { ToolLoopAgent, tool, stepCountIs } from 'ai'
import { openai } from '@ai-sdk/openai'
import { z } from 'zod'

// 注意：AI SDK 6 用 "provider/model" 字符串格式
const weatherAgent = new ToolLoopAgent({
  model: "openai/gpt-4o-mini",
  system: '你是天气助手，用中文回答问题',

  tools: {
    getWeather: tool({
      description: '获取城市天气',
      inputSchema: z.object({       // ⚠️ v6 用 inputSchema，不是 parameters
        city: z.string(),
      }),
      execute: async ({ city }) => ({
        city,
        temp: 22,
        condition: '晴',
      }),
    }),
  },

  stopWhen: stepCountIs(10), // 最多执行 10 步
})

// 使用：generate（等待完整结果）
const result = await weatherAgent.generate({
  prompt: '北京今天天气怎么样？',
})
console.log(result.text)
console.log(result.steps)   // 所有执行步骤
console.log(result.usage)   // token 用量汇总
流式模式 — stream()
在 Next.js API Route 中使用
// app/api/agent/route.ts
export async function POST(req: Request) {
  const { prompt } = await req.json()

  // stream() 返回 ReadableStream
  const result = weatherAgent.stream({ prompt })

  return result.toDataStreamResponse()
}

// 前端仍然用 useChat，与普通 streamText 完全兼容
result.steps — 查看 Agent 的推理过程
steps 的结构
const result = await agent.generate({ prompt })

for (const step of result.steps) {
  console.log('步骤类型:', step.stepType)
  // 'initial' | 'tool-result' | 'continue'

  console.log('工具调用:', step.toolCalls)
  // [{ toolName: 'getWeather', args: { city: '北京' } }]

  console.log('工具结果:', step.toolResults)
  // [{ toolName: 'getWeather', result: { temp: 22 } }]

  console.log('本步骤文本:', step.text)
  console.log('Token用量:', step.usage)
}
⚠️ v5 → v6 迁移注意：inputSchema 替换 parameters

v5 写法（旧）

tool({
  description: '...',
  parameters: z.object({
    city: z.string(),
  }),
  execute: async (args) => ...
})

v6 写法（新）

tool({
  description: '...',
  inputSchema: z.object({
    city: z.string(),
  }),
  execute: async (args) => ...
})
迁移工具：运行 npx @ai-sdk/codemod v6 可以自动把旧写法批量升级到 v6 语法。

Chapter 03
Loop Control
精确控制循环

Agent 默认会一直循环到认为任务完成为止。Loop Control 让你精确定义"什么时候停"和"每步之前做什么"，避免无限循环和失控执行。

两个核心 API
stopWhen prepareStep

stopWhen 定义终止条件，prepareStep 在每步执行前插入逻辑（动态修改工具、注入上下文等）。

stopWhen — 内置终止条件
import {
  ToolLoopAgent,
  stepCountIs,       // 执行了 N 步就停
  hasToolCall,       // 调用了某个工具就停
  hasNoToolCalls,    // 没有调用工具就停（AI 认为完成）
} from 'ai'

// 示例 1：最多跑 5 步（防止失控）
new ToolLoopAgent({
  model: 'openai/gpt-4o-mini',
  tools: { ... },
  stopWhen: stepCountIs(5),
})

// 示例 2：没有工具调用时停止（AI 认为任务完成）
new ToolLoopAgent({
  stopWhen: hasNoToolCalls(), // 这也是默认行为
})

// 示例 3：组合条件 — 满足任一即停
new ToolLoopAgent({
  stopWhen: [
    stepCountIs(10),     // 超过 10 步停止
    hasToolCall('done'), // 或调用了"完成"工具
  ],
})
自定义 stopWhen — 根据业务逻辑停止
import { StopCondition } from 'ai'

// stopWhen 接收一个函数：(stepResult) => boolean
const agent = new ToolLoopAgent({
  model: 'openai/gpt-4o-mini',
  tools: { getBalance, transfer, notify },

  stopWhen: (stepResult) => {
    // 如果 AI 调用了"发送通知"工具，说明已完成
    return stepResult.toolCalls.some(
      c => c.toolName === 'notify'
    )
  },
})
prepareStep — 每步执行前插入逻辑
最强大的功能之一：在每个步骤开始前，你可以动态修改工具集、注入新上下文、切换模型，实现高度灵活的 Agent 行为。 
const agent = new ToolLoopAgent({
  model: 'openai/gpt-4o-mini',
  tools: { search, analyze, writeReport },

  prepareStep: async ({ stepNumber, messages, toolCalls }) => {
    // 前 3 步只能用 search 工具
    if (stepNumber < 3) {
      return {
        toolChoice: { type: 'tool', toolName: 'search' },
      }
    }

    // 如果有足够多的搜索结果，切换到分析模式
    const searchCount = toolCalls.filter(c => c.toolName === 'search').length
    if (searchCount >= 3) {
      return {
        // 动态注入额外上下文
        system: '你已收集了足够信息，现在请撰写分析报告',
        toolChoice: { type: 'tool', toolName: 'writeReport' },
      }
    }

    return {} // 不修改，正常执行
  },
})
onStep 回调 — 实时观察每步结果
const result = await agent.generate({
  prompt: '分析竞争对手产品',

  // 每步完成后触发（不阻塞执行）
  onStep: (step) => {
    console.log(`第 ${step.stepNumber} 步完成`)
    console.log('调用工具:', step.toolCalls.map(c => c.toolName))

    // 可以把进度推送给前端
    progressChannel.send({
      step: step.stepNumber,
      action: step.toolCalls[0]?.toolName,
    })
  },
})

Chapter 04
Memory
Agent 的记忆

没有记忆的 Agent 每次都是全新开始。Memory 让 Agent 能记住用户偏好、历史操作、长期知识，是构建个性化 AI 应用的关键。

记忆的三种形态
1
短期记忆（In-Context Memory）
当前对话的 messages 数组。随对话结束消失。AI SDK 自动管理。
2
外部记忆（External Memory）
存进数据库（Postgres/Redis）。跨对话持久化。需要你自己实现存取逻辑。
3
语义记忆（Semantic / RAG）
把历史内容向量化存储，按语义检索相关片段注入上下文。即第二篇教程的 Embedding 技术。
模式 1：持久化对话历史（跨会话短期记忆）

核心思路：每次对话前从 DB 加载历史，对话结束后存回 DB。

// lib/memory.ts
import { db } from './db' // 你的数据库客户端

export async function loadHistory(userId: string) {
  const rows = await db.query(
    `SELECT role, content FROM messages
     WHERE user_id = $1
     ORDER BY created_at DESC
     LIMIT 20`,
    [userId]
  )
  return rows.reverse() // 按时间正序
}

export async function saveMessages(
  userId: string,
  messages: CoreMessage[]
) {
  await db.query(
    `INSERT INTO messages (user_id, role, content)
     VALUES ($1, $2, $3)`,
    messages.map(m => [userId, m.role, m.content])
  )
}
在 Agent 中使用
// app/api/chat/route.ts
export async function POST(req: Request) {
  const { userId, message } = await req.json()

  // 1. 加载历史对话
  const history = await loadHistory(userId)

  // 2. 运行 Agent（带历史上下文）
  const result = await myAgent.generate({
    messages: [
      ...history,                           // 历史消息
      { role: 'user', content: message },  // 新消息
    ],
  })

  // 3. 存储新消息和 AI 回复
  await saveMessages(userId, [
    { role: 'user', content: message },
    { role: 'assistant', content: result.text },
  ])

  return Response.json({ reply: result.text })
}

// 每次对话结束后：提取并存储用户信息
async function updateUserProfile(
  userId: string,
  conversation: CoreMessage[]

  // 用 generateObject 从对话中提取用户偏好
  const { object: profile } = await generateObject({

      name: z.string().optional(),
      preferredLanguage: z.string().optional(),

模式 3：语义记忆 + RAG（最强大）
组合使用：把第二篇教程学的 embed() + 向量数据库，用来存储和检索历史对话中的重要信息片段。 
// 存储记忆：把重要信息向量化存入数据库
async function storeMemory(userId: string, text: string) {
  const { embedding } = await embed({
    model: openai.embedding('text-embedding-3-small'),
    value: text,
  })
  await vectorDB.upsert({ userId, text, embedding })
}

// 检索记忆：找出与当前问题最相关的历史记忆
async function retrieveMemory(userId: string, query: string) {
  const { embedding } = await embed({
    model: openai.embedding('text-embedding-3-small'),
    value: query,
  })
  return vectorDB.query({ userId, embedding, topK: 5 })
}

// 使用：构建带相关记忆的 Agent 调用
const memories = await retrieveMemory(userId, userMessage)
const result = await myAgent.generate({
  system: `你是智能助手。以下是与此次对话相关的历史记忆：
${memories.map(m => `- ${m.text}`).join('\n')}`,
  prompt: userMessage,
})

Chapter 05
Subagents
Agent 调 Agent

复杂任务用单个 Agent 难以胜任。Subagent 模式让一个 Orchestrator Agent 把子任务委托给专门的 Expert Agent，实现分工协作。

三种多 Agent 架构
A
顺序委托（Sequential）
Orchestrator → Agent A 完成 → Agent B 完成 → Agent C 完成 → 汇总
B
并行委托（Parallel）
Orchestrator 同时启动多个 Agent，用 Promise.all 等待全部完成再合并结果
C
动态路由（Router）
Orchestrator 根据问题类型，动态选择最合适的专家 Agent 处理
核心思路：把 Agent 包装成 Tool
关键设计：AI SDK 的 Subagent 模式非常优雅 — 把一个 Agent 的 generate() 调用封装进 tool() 的 execute 里，让 Orchestrator 像调工具一样调用子 Agent。
并行执行 Subagents（提升效率）
动态路由 Agent — 根据问题分流

Chapter 06
Multi-Agent
网络协作

多个 Agent 组成协作网络，每个 Agent 独立运行、相互通信，共同完成单个 Agent 无法胜任的复杂任务。这是 AI SDK 对 Subagent 概念的进一步延伸。

⚡ Subagent vs Multi-Agent — 核心区别

SUBAGENT 模式

↓
层级制（Hierarchical）
有明确的 Orchestrator（主 Agent）主动调用 Subagent（从 Agent）。控制权在上层。
🔧
Subagent 被封装成 Tool
Orchestrator 通过 Tool Call 触发子 Agent，子 Agent 的执行对主 Agent 是透明的。
📋
适合：有中心协调者的任务
如"报告生成"：主 Agent 先调研再写作，流程固定、有主控方。

MULTI-AGENT 模式

⇄
网状制（Networked）
多个平等的 Agent 各自独立运行，通过消息/共享状态互相通信，无单一控制中心。
📡
Agent 通过消息传递协作
每个 Agent 可以向其他 Agent 发消息、等待回复，形成真正的协作网络。
🌐
适合：需要多方并行协作的任务
如"代码审查流水线"：多个专家 Agent 并行工作，互相 review 结果。
架构图对比
SUBAGENT — 树状层级
Orchestrator Agent

MULTI-AGENT — 网状协作

⇅ ⇅

AI SDK 如何支持 Multi-Agent？
关键理解：AI SDK 本身不内置"Multi-Agent 框架"，但它提供了构建多 Agent 协作所需的全部原语。你用 ToolLoopAgent + 异步通信模式来实现 Multi-Agent 网络。
ToolLoopAgent × N Promise.all 并行 消息队列 共享状态 事件驱动
模式 1：并行网络（最常用）

多个专家 Agent 同时运行，结果汇总给 Synthesizer Agent 整合。

import { ToolLoopAgent, tool, stepCountIs } from 'ai'

// ── 定义专家 Agent 网络 ──
const agents = {
  frontend: new ToolLoopAgent({
    model: 'openai/gpt-4o',
    system: '你是前端架构专家，专注 React/TypeScript/性能优化',
    tools: { searchDocs, runLinter },
  }),

  backend: new ToolLoopAgent({
    model: 'openai/gpt-4o',
    system: '你是后端架构专家，专注 API 设计/数据库/安全',
    tools: { searchDocs, queryDB },
  }),

  security: new ToolLoopAgent({
    model: 'anthropic/claude-sonnet-4-5', // 可以混用不同模型！
    system: '你是安全专家，专注漏洞识别/OWASP/渗透测试',
    tools: { scanVulnerabilities, checkDependencies },
  }),

  synthesizer: new ToolLoopAgent({
    model: 'openai/gpt-4o',
    system: '你是技术总监，负责整合多个专家意见给出最终方案',
  }),
}

// ── 并行执行三个专家，再汇总 ──
async function codeReview(prDescription: string) {
  // Step 1: 三个 Agent 同时分析（节省 2/3 时间）
  const [frontendReview, backendReview, securityReview] =
    await Promise.all([
      agents.frontend.generate({
        prompt: `审查这个 PR 的前端部分：\n${prDescription}`,
      }),
      agents.backend.generate({
        prompt: `审查这个 PR 的后端部分：\n${prDescription}`,
      }),
      agents.security.generate({
        prompt: `检查这个 PR 的安全风险：\n${prDescription}`,
      }),
    ])

  // Step 2: Synthesizer 整合所有意见
  const finalReview = await agents.synthesizer.generate({
    prompt: `整合以下三位专家的 Code Review 意见，给出优先级排序和最终建议：

【前端专家】
${frontendReview.text}

【后端专家】
${backendReview.text}

【安全专家】
${securityReview.text}`,
  })

  return {
    final: finalReview.text,
    details: { frontendReview, backendReview, securityReview },
    totalSteps: frontendReview.steps.length
      + backendReview.steps.length
      + securityReview.steps.length,
  }
}

模式 3：共享状态（Agent 读写同一个上下文）

多个 Agent 共享一个"项目状态"对象，通过读写状态来协调行动。

选择指南：什么时候用哪种模式？
S
用 Subagent — 有明确主控方的任务
任务流程相对固定，需要一个"大脑"统一决策。如：报告生成、客服路由、内容审核流水线。
M
用 Multi-Agent — 需要平等协作的任务
多个专业领域需要同等权重的输入，没有单一的正确流程。如：代码评审、投资分析、研究综述。
↑
混合使用 — 最常见的生产方案
外层用 Subagent 模式做任务分发，内层用 Multi-Agent 并行处理。两种模式可以自由组合嵌套。
